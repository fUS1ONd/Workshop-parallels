# Workshop-parallels

Все задачи строятся вокруг свёртки изображений ([Convolution](https://lodev.org/cgtutor/filtering.html)). Для простоты достаточно реализовать работу с простой битмапой в градациях серого (цвветные картинки для сильных духом). Для каждой задачи надо выполнить анализ производительности хотя бы в виде графиков в ридми к репо и краткого анализа.

**Ограничения**

- Языки. На выбор C, JVM (Java/Kotlin), .NET (C\#/F\#).
- Для простоты можно воспользоваться какой-нибудь библиотекой для загрузки изображения, его конвертации в градации серого и сохранения на диск. Например, для всех допустимых стеков есть OpenCV, но можно взять и что-то специфическое для стека (например [https://github.com/SixLabors/ImageSharp](https://github.com/SixLabors/ImageSharp) для .NET)

**Дедлайны (2025-2026 учебный год)**

- 1 и 2 задачи должны быть сданы до 30 апреля включительно
- 3 задача должна быть сдана до 15 мая включительно
- При наличии задолженности (пересдачи/комиссии) или индивидуального графика задачи должны быть отправлены не позднее чем за 2 недели до даты зачета
- Дополнительные задачи принимаются не позднее 2 недель до даты зачета

**Задачи**

1. Последовательная реализация для одного изображения.
2. Параллельная реализация для одного изображения.
   1. Разные способы разделения изображения на части. Позволяет понаблюдать эффекты от доступа к данным.
      1. Попиксельно
      2. По строкам
      3. По столбцам
      4. Произвольная прямоугольная решётка
   2. Можно поиграться с композицинальностью фильтров, их последовательным применением, размерами кэша.
3. Разделение чтения, записи, свёртки, для потоковой обработки массива изображений, что можно разными способами делать
   1. Если чтение успевает, то оно может прокормить несколько свёрток.
   2. При этом, сами свёртки можно параллелить или нет.
   3. Балансировка нагрузки: если чтение шустрое, а свёртка тупит, но надо следить, чтобы ридер всю память не забил. А ещё запись может тупить.
4. Свёртка на GPU. (Для сильных духом 1\)
   1. Просто перенести свёртку на ГПУ
   2. Раскидать задачи по свёртке на ГПУ и CPU.
   3. Попробовать пооптимизировать свёртку на ГПУ.
   4. На самом деле, на одной ГПУ иногда можно держать несколько свёрток.
   5. Рекомендованные библиотеки
      1. Для С проблем нет (CUDA и OpenCL должны адекватно работать)
      2. Для .NET [ILGPU](https://ilgpu.net/) или [Brahma.FSharp](https://www.nuget.org/packages/Brahma.FSharp/)
      3. Для JVM вроде рекомендуют [https://jogamp.org/](https://jogamp.org/), конкретно [JOCL](https://jogamp.org/jocl/www/)
5. Распределённая свёртка. (Для сильных духом 2\)
   1. Простой вариант: каждый узел обрабатывает изображение целиком, но изображения раскидываются по узлам.
   2. Чуть более сложный: несколько узлов обрабатывают одну большую картинку. Тут сложнее, так как эту картинку разделить, а потом аккуратно собрать обратно результат.
6. Можно провести соревнования на нашем железе, чтобы все в одном окружении.

**Схема тестирования**

1. Если библиотека работы с изображениями поддерживает свёртку, то она может быть референсом для последовательной реализации
2. Последовательная реализация может быть референсом.
   1. Для неё, кроме ручных тестов с эталонной картинкой, можно проверить кучу свойств и это, в целом, даёт неплохую надёжность. Надо только не забывать про разную обработку границ.
      1. Фильтры композициональны. Потому последовательное применение нескольких случайных фильтров к случайной картинке — то же самое, что применение композиции этих фильтров.
      2. Можно придумать фильтры, которые в композиции дают id: например, сдвиг сперва в одну сторону, потом в другую.
      3. Если фильтр расширить нулями, то результат не изменяется.
      4. Есть простые фильтры, результат применения которых известен для любого изображения. Например, нулевой или id (единичка в центре — остальные нули).
      5. При таких тестах, обязательно поперебирать размеры изображений и фильтров в достаточно широких границах.
      6. Ещё можно проверить свойство, что реализация ведёт себя также, как из какой-нибудь библиотеки. Но для этого надо подобрать библиотеку.
3. Для остальных реализаций проверяем банальное свойство: на случайных данных (изображение и фильтр) любая параллельная версия ведёт себя точно так же, как и последовательная.
4. Можно попытаться при тестировании поиспользовать разные инструменты обнаружения ошибок многопоточности (например, TSan, Lincheck, и др.)
